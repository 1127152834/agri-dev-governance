# 团队开发规范

## 1. 定位

本规范用于定义团队开发前的统一门禁规则。任何成员或 AI 助手在开始需求分析、产品设计、实施计划、开发实现、测试验证、提交流程前，都必须先经过本规范检查，确认当前任务已满足进入下一阶段的前置条件。

## 2. 适用范围

适用于智慧农业团队所有项目仓库，包括但不限于物联网、数据中台、业务系统及其前后端子仓库。

## 3. 核心职责

团队开发规范只负责以下事项：

- 判断当前任务所处阶段
- 检查当前阶段是否满足门禁
- 列出缺失文档或缺失前置条件
- 指引下一步应该调用哪个 Skill
- 明确当前阶段禁止事项

## 4. 阶段判断

### 4.1 任务提出 / 拆分

如果当前只有业务目标、问题背景、优化建议，而尚未形成明确任务项，则处于任务提出或任务拆分阶段。

### 4.2 需求说明

如果任务已认领，但尚未形成 `01-需求说明.md`，则处于需求说明准备阶段。

### 4.3 产品设计

如果需求说明已完成，但尚未形成 `02-产品设计.md`，则处于产品设计阶段。

### 4.4 实施计划

如果产品设计已完成，但尚未形成 `03-实施计划.md`，则处于实施计划阶段。

### 4.5 开发实现

如果实施计划已完成，且已创建独立 worktree，则可进入开发实现阶段。

### 4.6 测试验证

如果代码已完成，但尚未形成 `05-测试报告.md`，则处于测试验证阶段。

### 4.7 提交与 PR

如果测试报告已完成，则可进入提交与 PR 阶段。

## 5. 门禁规则

### 5.1 进入需求说明前

- 任务必须已发布、已认领或已指派
- 责任人必须明确

### 5.2 进入产品设计前

- 必须存在 `01-需求说明.md`
- 需求目标、范围和验收标准必须明确

### 5.3 进入实施计划前

- 必须存在 `02-产品设计.md`
- 关键流程、边界和异常场景必须明确

### 5.4 进入开发实现前

- 必须存在 `03-实施计划.md`
- 必须创建独立 worktree
- 必须知道当前任务对应的文档目录

### 5.5 进入提交与 PR 前

- 必须有测试验证结果
- 必须补齐相关文档引用
- 必须满足提交规范

## 6. 推荐文档路径

模块开发默认文档路径：

```text
docs/
└── 成员名/
    └── 模块名称/
        ├── 01-需求说明.md
        ├── 02-产品设计.md
        ├── 03-实施计划.md
        ├── 04-开发记录.md
        ├── 05-测试报告.md
        ├── 06-发布说明.md
        └── 07-复盘总结.md
```

## 7. Skill 分流

| 当前阶段 | 推荐 Skill |
|----------|------------|
| 需求说明 | `product-strategist` |
| 产品设计 | `product-strategist` / `product-designer` |
| 实施计划 | `writing-plans` |
| 创建 worktree | `using-git-worktrees` |
| 开发实现 | `test-driven-development` |
| 测试验证 | `verification-before-completion` |
| 提交与 PR | `commit-pr-spec` |

## 8. 禁止事项

- 禁止没有需求说明就直接开发
- 禁止没有实施计划就直接编码
- 禁止不使用 worktree 就进行正式功能开发
- 禁止没有测试结果就提交 PR
- 禁止跳过文档链路直接宣称任务完成
