# 提交与 PR 规范

## 1. 定位

本规范用于统一团队的提交与 PR 质量门禁，确保每次代码交付都具备可读、可审、可验证、可回滚的基本条件。

## 2. commit message 规范

推荐使用以下前缀：

- `feat`
- `fix`
- `refactor`
- `docs`
- `chore`
- `test`

推荐格式：

```text
type: 简洁说明本次改动
```

示例：

```text
feat: add task claim skill
fix: correct lifecycle viewport zoom bounds
```

## 3. PR 必填项

每个 PR 至少要包含：

- 改动摘要
- 关联模块
- 关联文档
- 测试证明
- 风险说明
- 回滚说明

## 4. 测试证明

测试证明至少要说明：

- 执行了什么验证
- 验证结果是什么
- 哪些场景未覆盖

## 5. 风险说明

必须明确：

- 可能影响的模块
- 可能产生的问题
- 是否有数据、状态或兼容性风险

## 6. 回滚说明

必须明确：

- 发生异常时如何回退
- 回滚影响范围
- 是否需要额外恢复动作

## 7. 禁止事项

- 禁止没有测试证明就提交 PR
- 禁止 PR 描述为空或只有一句话
- 禁止不写风险说明和回滚说明
- 禁止 commit message 含糊不清
