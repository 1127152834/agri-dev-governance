把团队协作流程做成可执行的操作系统

这不是一套写在文档里的制度，而是一套真的能跑起来的团队协作机制。 我们今天要讲的，是如何把项目初始化、任务池、认领、门禁、交付和日报串成一条可执行链。

40 分钟 认知 + 实操 设备管理模块主线
这场培训会回答什么
培训对象

项目经理、开发人员、使用 Codex 的成员

培训目标

建立统一认知，并让团队马上知道如何在真实项目里落地这套治理体系。

主线场景

设备管理模块：从项目初始化到日报沉淀，跑通一次完整闭环。

问题背景

团队项目真正失控的地方，不在技术，而在协作链路断裂

任务

来源混乱，口头沟通代替任务池

文档

需求、设计、计划彼此断链

交付

commit 和 PR 缺失测试证明与风险说明

状态

任务池、成员视图、模块状态彼此失真

今天要解决的问题

我们要把“项目怎么开始、任务怎么进入、开发怎么推进、交付怎么检查、状态怎么同步、日报怎么沉淀” 全部变成一套可重复执行、可追溯、可回滚的团队操作系统。

治理地图

当前主分支已经具备的治理能力，不是想法，而是可执行底座

  • `task-pool-manager`

关键认知

这套仓库的本质，不是把文档堆起来，而是把项目协作中最容易失控的动作做成有门禁、有记录、有出口的工作流。

整体闭环

真正要记住的，不是文件名，而是这条完整协作链

项目初始化 → 成员加入 → 任务进入 → 任务认领 → 开发前门禁 → 开发实现 → 提交交付 → 状态同步 → 每日日报
主记录

任务池永远是主记录。

派生视图

成员 `tasks.md`、`status.md`、模块文档目录只是派生视图。

门禁原则

没有前置文档和验证证据，不进入下一阶段。

角色分工

AI 助手 / Codex

  • 做门禁检查
  • 做状态同步与归档

步骤一 · 战术卡
01

项目初始化

第一步不是写代码，而是先把项目仓库变成一个能跑治理流程的操作环境。

为什么必须有这一步

如果新仓库没有任务池、项目级规则入口和 PR 模板，后续所有协作都会重新退回口头沟通和临时约定。项目初始化的本质，是先把“治理骨架”铺进去，再开始承接真实业务。

谁负责 / 怎么做

项目经理或项目负责人执行。

使用 $project-bootstrap 为一个新项目仓库初始化治理目录和规则入口。
会产出什么
  • `docs/project-overview.md`
  • `docs/task-pool/` 生命周期目录
  • 项目级 `AGENTS.md`
  • PR 模板
常见错误
  • 直接在空仓库开始开发
  • 项目没有任务池
  • 项目规则只存在口头说明里
步骤二 · 战术卡
02

成员加入

先建立个人工作容器，再承接真实任务，后续状态和日报才有统一落点。

为什么必须有这一步

没有 `profile.md`、`status.md`、`tasks.md` 和日报目录，状态同步和沉淀都无从谈起。

谁负责 / 怎么做

开发人员自己执行，AI 助手可协助生成。

使用 $member-init 初始化当前成员的资料、任务状态文件和日报目录。
会产出什么
  • `profile.md`
  • `status.md`
  • `tasks.md`
  • `reports/daily/README.md`
常见错误
  • 没做成员初始化就认领任务
  • 个人状态不落文档
  • 日报目录缺失
步骤三 · 战术卡
03

任务先进入任务池，再被认领

这一步决定了任务是不是能被真正追踪，能不能在不同角色之间稳定流转。

$task-pool-manager

把任务发布到 `01-ready`，指派后进入 `02-assigned`，并同步写入任务日志。

任务池是主记录
$task-claim

把任务同步到成员 `tasks.md`、`status.md` 和模块目录入口。

成员视图只是派生视图
为什么必须这样设计

如果任务先进入个人视图、后补任务池，团队很快就会失去统一真相源。

常见错误
  • 认领后删除任务文件
  • 只改成员视图，不改任务池
  • 不写任务流转日志
步骤四 · 战术卡
04

开发前门禁

这一步的意义，就是阻止“想写代码就直接写”的习惯进入团队协作链。

核心能力

`$team-dev-spec` 负责判断当前阶段、列出缺失项、指引下一步 Skill。

使用 $team-dev-spec 检查我当前任务是否满足进入下一阶段的开发门禁。
为什么必须做

它是整个体系里最重要的防线，防止文档断链、跳过设计、跳过计划、跳过验证。

典型门禁
  • 没有 `01-需求说明.md`，不能直接开发
  • 没有 `03-实施计划.md`，不能直接编码
  • 没有测试结果，不能进入提交与 PR
常见错误
  • 用口头说明替代文档
  • 把“应该没问题”当作验证
  • 跳过 worktree 直接开发
步骤五 · 战术卡
05

提交与交付

交付不是“把代码推上去”，而是让别人有足够信息判断这次改动是否可信。

核心能力

`$commit-pr-spec` 负责检查 commit message、PR 描述、测试证明、风险说明和回滚说明。

PR 最低门槛
  • 改动摘要
  • 关联文档
  • 测试证明
  • 风险说明
  • 回滚说明
为什么必须做这一步

如果交付描述不完整，评审只能靠猜，风险也无法真正被看见。

常见错误
  • PR 只写一句话
  • 没有测试证明
  • 不写回滚说明
步骤六 · 战术卡
06

状态同步与日报

这两步不是收尾动作，而是把团队协作真正闭合的最后两道机制。

$task-status-sync

对齐任务池、成员 `tasks.md`、成员 `status.md` 和模块文档状态。

以任务池状态为准
$daily-report

读取 `tasks.md` 与 `status.md`，生成个人日报草稿，再由成员补充阻塞项和明日计划。

日报是沉淀，不是形式
为什么必须做这一步

如果状态不同步、日报不沉淀，团队第二天就会失去上下文，协作成本会急剧上升。

常见错误
  • 任务状态改了，但成员面板不更新
  • 只补日报，不同步 `status.md`
  • 日报只写情绪，不写任务和风险

错误警示

大多数团队不是因为不会写代码，而是因为总在重复这些协作错误

没做 member-init 就认领任务

后续状态和日报没有落点。

没过 team-dev-spec 就直接开发

最容易导致文档断链和返工。

只改成员视图，不改任务池

主记录和派生视图失真，后续状态会越滚越乱。

没有测试证明就发 PR

评审人无法判断交付是否可信。

会后行动

培训结束后，最重要的不是记住文件名，而是开始在真实项目里跑起来

建议落地顺序
01
挑一个真实项目仓库

先接入 `project-bootstrap`，不要从抽象演练开始。

02
让一个 PM + 一个开发先试跑

先小范围试点，再向全团队推广。

03
按周回看规则是否过重

不是越多越好，而是越稳定越好。
