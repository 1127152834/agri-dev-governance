# 工程手册

## 1. 目标与适用范围

本手册用于定义智慧农业团队的统一研发治理规则，适用于组织内所有项目仓库，包括但不限于物联网、数据中台、业务系统及其前后端子仓库。

目标是建立一套同时约束人工成员与 AI 助手的研发流程，保证多仓协作下的开发活动具备一致性、可追溯性、可验证性与可审计性。

## 2. 多仓协作原则

- 所有项目都必须遵循统一的组织级工程规范。
- 每个业务仓库可以补充局部规则，但不能与组织级规则冲突。
- 所有仓库都必须要求 Git 协作，不允许脱离版本管理进行长期开发。
- 所有模块开发都必须有对应文档沉淀，文档和代码必须互相可追溯。

## 3. 强制开发主流程

统一流程如下：

1. 成员初始化
2. 需求登记
3. 产品设计
4. 实施计划
5. 创建 worktree
6. 开发实现
7. 测试验证
8. 提交与 PR
9. 评审合并
10. 文档回写与复盘

任何阶段如未满足前置条件，不允许进入下一阶段。

## 4. 文档追溯规范

每项模块开发必须在对应仓库中建立统一目录，推荐结构如下：

```text
docs/
└── 成员姓名/
    └── 模块名称/
        ├── 01-需求说明.md
        ├── 02-产品设计.md
        ├── 03-实施计划.md
        ├── 04-开发记录.md
        ├── 05-测试报告.md
        ├── 06-发布说明.md
        └── 07-复盘总结.md
```

成员个人目录建议保留以下文件：

```text
docs/
└── 成员姓名/
    ├── profile.md
    ├── tasks.md
    └── status.md
```

## 5. Git / worktree / 提交 / PR 原则

- 所有开发必须基于 Git。
- 所有功能开发默认使用独立分支，禁止直接在主分支上开发。
- 默认要求使用 `git worktree` 隔离并行任务，避免多任务污染同一工作区。
- 提交信息必须符合统一提交规范。
- PR 必须关联对应文档与测试结论。
- 未经评审通过，不允许合并到主分支。

## 6. 规则建设原则

- 规范优先沉淀到组织治理仓库，再扩散到业务仓库。
- 能用模板统一的，不靠口头约定。
- 能用脚本校验的，不只靠人工检查。
- 能抽象为 Skill 的流程，必须有明确输入、输出、门禁和产物。
